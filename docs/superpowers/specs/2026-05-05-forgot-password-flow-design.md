# 忘记密码接入设计

## 背景

当前系统已经具备账号注册、登录、会话维持、多用户隔离和系统级 SMTP 配置能力，数据库中也已存在 `PasswordResetToken` 模型与对应关联关系。

但用户从登录页进入“找回密码”后，前后端仍然是占位状态：

- 前端已存在 `/forgot-password` 与 `/reset-password` 路由和页面壳子，但不能提交。
- 后端仅提供注册、登录、退出接口，没有忘记密码和重置密码接口。
- 邮件服务配置已经接入系统设置，但认证邮件与重置密码邮件仍是占位发送实现。

这会直接阻断用户自助恢复登录能力，也会让登录页的入口变成无效承诺。

## 目标

补齐一条可用的密码找回链路，让用户能够：

1. 在登录页点击“忘记密码”进入找回页面。
2. 输入注册邮箱后提交找回请求。
3. 在邮箱中收到重置密码入口。
4. 通过入口进入重置密码页并设置新密码。
5. 重置成功后返回登录页重新登录，继续使用自己原有的私有创作空间。

## 非目标

本次不处理以下范围：

- 不新增短信验证码或双因素验证。
- 不调整注册、邮箱验证、会话模型的整体策略。
- 不引入公开的系统邮件配置页面给普通用户，系统 SMTP 仍然只由管理员配置。
- 不扩展为管理员代用户重置密码。
- 不修改桌面端相关流程。

## 现状约束

- 系统邮件配置是系统级资源，普通用户不可见。
- 所有账号空间保持私有，各自使用自己的数据空间。
- 数据库以 PostgreSQL 为主，SQLite 不作为实际生产运行目标，但现有测试仍可能覆盖 SQLite schema。
- 用户是写作新手，交互应尽量直接，避免暴露技术细节。
- UI 文案必须从用户视角描述动作和结果，不写成开发过程说明。

## 方案对比

### 方案 A：发送带明文 token 的重置链接，服务端只保存哈希值

推荐采用。

优点：

- 与现有邮箱验证 token 设计一致，最容易复用当前认证体系。
- 数据库泄露时，重置 token 明文不会直接暴露。
- 接口边界清晰，前端只需要接 token 链接并提交新密码。

缺点：

- 需要补齐 token 生成、过期校验、单次消费逻辑。

### 方案 B：发送短验证码，用户手动输入验证码和新密码

不推荐本次采用。

优点：

- 链接依赖更低。

缺点：

- 用户操作步骤更多，不利于降低认知负担。
- 前端需要额外输入验证码流程。
- 对当前已有 `/reset-password` 路由帮助不大。

### 方案 C：只允许管理员后台触发重置

不推荐。

优点：

- 实现简单。

缺点：

- 不满足用户自助恢复账号的核心需求。
- 与登录页已暴露的“忘记密码”入口不匹配。

## 推荐方案

采用方案 A，新增两条认证接口并把前端占位页面改成实际表单：

- `POST /api/auth/forgot-password`
- `POST /api/auth/reset-password`

邮件中发送带 token 的网页入口，目标地址为前端的 `/reset-password?token=...`。

## 详细设计

### 1. 后端接口

#### 1.1 忘记密码

新增 `POST /api/auth/forgot-password`。

请求体：

- `email`

行为：

1. 归一化邮箱。
2. 无论邮箱是否存在、账号是否可用，对外都返回统一成功响应，避免枚举邮箱。
3. 当邮箱对应可用用户时：
   - 删除该用户此前未使用或未过期的旧重置 token。
   - 生成新的 opaque token。
   - 只保存 token 哈希值与过期时间。
   - 发送包含重置链接的邮件。
4. 返回统一提示，例如“如果该邮箱已注册，我们会把重置入口发送到你的邮箱”。

账号筛选规则：

- `disabled` 账号不发送重置邮件。
- 未找到账号时静默成功。

#### 1.2 重置密码

新增 `POST /api/auth/reset-password`。

请求体：

- `token`
- `password`

行为：

1. 校验密码强度。
2. 通过 token 哈希查找记录。
3. 拒绝以下情况：
   - token 不存在；
   - token 已过期；
   - token 已被消费；
   - 关联用户不存在或已停用。
4. 通过校验后：
   - 更新用户密码哈希；
   - 标记当前 token 已使用；
   - 清理该用户其他未使用的重置 token；
   - 删除该用户所有现有登录 session，强制旧设备重新登录。
5. 返回成功消息，不直接登录。

### 2. 邮件发送

复用现有系统级 SMTP 配置服务。

需要补齐认证邮件发送模块中的实际发送能力：

- 从系统设置读取 SMTP 主机、端口、账号、密码、发件人等配置。
- 生成用户可点击的重置链接。
- 发送标题明确、动作导向的重置密码邮件。

邮件内容要求：

- 说明这是重置登录密码入口。
- 提示入口有效期。
- 告知如果不是本人操作可以忽略。

为了兼顾前后端部署差异，重置链接基础地址优先按以下顺序解析：

1. 显式前端公开地址环境变量。
2. 通用站点公开地址环境变量。
3. 请求来源推导出的站点地址，作为开发和测试兜底。

### 3. 前端页面

#### 3.1 找回密码页

将占位页改为真实表单：

- 一个邮箱输入框。
- 一个提交按钮。
- 一个“返回登录”入口。

提交成功后不暴露账号是否存在，只展示统一提示，引导用户查看邮箱。

#### 3.2 重置密码页

将占位页改为真实表单：

- 从 URL 读取 `token`。
- 一个新密码输入框。
- 一个确认新密码输入框。
- 一个提交按钮。

交互规则：

- 缺少 token 时直接提示入口无效，并提供返回登录入口。
- 两次密码不一致时前端先阻止提交。
- 成功后展示成功提示，并提供返回登录按钮。
- token 无效或过期时展示明确的重新获取入口。

### 4. 安全与隔离

- 忘记密码接口响应统一，避免邮箱探测。
- token 使用哈希存储，不保存明文。
- token 必须有失效时间，并且单次使用。
- 重置密码后注销所有旧 session，防止旧登录态继续可用。
- 本流程不依赖用户传入 tenantId，也不改变现有私有资源归属。

### 5. 错误处理

对用户可见的错误分为三类：

1. 表单错误
   - 邮箱格式不正确
   - 密码长度不足
   - 两次密码不一致

2. 重置入口错误
   - 重置入口无效
   - 重置入口已过期

3. 系统错误
   - 邮件服务不可用
   - 系统繁忙，请稍后再试

忘记密码接口的系统错误需要权衡：

- 如果邮箱存在且发送失败，接口应返回失败，让用户知道邮件没有真正发出。
- 但对于不存在账号，仍然返回统一成功。

### 6. 测试策略

先补测试，再写实现。

后端至少覆盖：

- 已注册用户提交忘记密码可生成 token 记录并返回成功。
- 未注册邮箱提交忘记密码也返回统一成功，且不生成记录。
- 有效 token 可成功重置密码。
- 无效 token 重置失败。
- 过期 token 重置失败。
- 重置成功后旧密码失效，新密码可登录。
- 重置成功后旧 session 失效。

前端至少覆盖或手动验证：

- 找回密码页可提交并展示统一提示。
- 重置密码页可读取 token。
- 无 token、密码不一致、无效 token 的提示正确。
- 重置成功后可返回登录。

## 模块边界

建议在现有认证模块内补齐，避免把找回密码逻辑散落到通用服务中：

- `server/src/routes/auth.ts`
  - 新增忘记密码与重置密码路由。
- `server/src/services/auth/AuthService.ts`
  - 新增忘记密码与重置密码核心逻辑。
- `server/src/services/auth/authMail.ts`
  - 补齐重置密码邮件实际内容与发送实现。
- `client/src/api/auth.ts`
  - 新增忘记密码与重置密码 API 调用。
- `client/src/pages/auth/ForgotPasswordPage.tsx`
  - 改造成真实提交表单。
- `client/src/pages/auth/ResetPasswordPage.tsx`
  - 改造成真实提交表单。

如果 `AuthService.ts` 因此继续增长并逼近架构阈值，应拆出专门的密码重置服务文件。

## 验收标准

- 登录页“忘记密码”入口可用。
- 用户提交注册邮箱后能收到重置入口邮件。
- 邮件入口能打开网页重置密码页。
- 用户成功设置新密码后，旧密码不可再登录。
- 重置后旧登录态失效。
- 不存在账号时，接口不会泄露邮箱是否注册。
- 错误提示对普通用户清晰直接。
