# 对象外部知识库绑定与自动生成 Spec 草案

## 1. 背景与目标

本任务的目标是为“小说对象”建立一套外部知识库绑定机制，让书本所需的关键创作资产可以从外部知识库中自动生成，再保存为本项目内的正式资产。

这里的“外部知识库”不是指本项目不保存结果，也不是把小说依赖永久放在远端。正确边界是：

- 外部知识库提供资料、证据、抽取结果、候选建议和生成依据。
- 本项目负责把这些结果转成世界观、人物、写法、反 AI、文笔等正式资产。
- 本项目保存最终资产、来源引用、生成版本、用户确认状态和后续编辑记录。
- 后续规划、写作、审校、修复默认消费本项目内的正式资产，而不是每次临时依赖外部知识库重新推断。

这条能力服务于新手成书主链：用户只需要绑定资料来源，系统自动补齐一套可继续推进整本小说的书级资产，用户以确认和少量修正为主。

## 2. 任务边界

### 2.1 本期要解决

1. 定义哪些书本信息可以由外部知识库辅助自动生成。
2. 明确每类信息在本项目内的正式资产形态。
3. 明确每类资产的生成方式、来源追溯、确认状态和后续消费边界。
4. 为后续跨系统接口设计列出上游必须支持的结构化能力。
5. 形成一个可评审的 spec，确认后再进入跨系统交互合同设计。

### 2.2 本期不解决

1. 不直接设计具体 HTTP API、鉴权、分页、错误码和 webhook。
2. 不直接实现代码或数据库迁移。
3. 不把外部知识库变成本项目的唯一事实源。
4. 不绕过本项目现有 Prompt Registry、资产服务、任务中心和审核确认链路。
5. 不新增关键词匹配式的资料分类或路由；资料理解、资产抽取和适配推荐必须走 AI 结构化输出。

## 3. 核心对象

### 3.1 NovelExternalKnowledgeBinding

小说与外部知识库来源之间的绑定对象。

建议字段：

- `novelId`：本项目小说 ID。
- `externalSourceId`：外部知识库来源 ID。
- `sourceType`：外部系统中的来源类型，例如作品库、资料集、拆书库、作者资料库、风格库。
- `sourceScope`：绑定范围，例如整套资料、某本书、某批章节、某个风格样本集。
- `sourceVersion`：上游资料版本或快照号。
- `enabledAssetTypes`：本次允许生成的资产类型。
- `status`：已绑定、生成中、需要确认、已应用、已失效。
- `createdBy` / `createdAt` / `updatedAt`。

### 3.2 GeneratedBookAssetDraft

外部知识库生成结果进入本项目后的草稿对象。它不是最终资产，而是待确认、待合并或待编辑的候选。

建议字段：

- `assetType`：世界观、人物、写法、反 AI、文笔等。
- `targetLevel`：书级、卷级、章节级、角色级、任务级。
- `draftPayload`：结构化生成结果。
- `evidenceRefs`：上游证据引用。
- `confidence`：上游和本项目综合判断的可信度。
- `coverage`：资料覆盖范围，避免局部资料被误当作整本结论。
- `generationMode`：全量生成、增量补齐、差异更新、冲突修复。
- `reviewStatus`：待确认、已应用、已忽略、需重生成。
- `appliedAssetId`：应用后对应的本项目资产 ID。

### 3.3 EvidenceRef

所有生成资产都必须能追溯来源。

建议字段：

- `sourceId` / `sourceVersion`。
- `documentId` / `segmentId` / `chapterRange`。
- `quoteSummary`：证据摘要，不要求保存长原文。
- `reason`：这条证据支持哪个判断。
- `coverageLabel`：例如“覆盖前 20 章”“覆盖样本文本 8 篇”“覆盖世界设定资料集”。

## 4. 资产范围

### 4.1 世界观

目标：从外部知识库中自动生成或补齐可用于本书的世界观切片，而不是生成百科式设定堆。

本项目正式资产建议落到：

- 世界概要。
- 世界规则。
- 势力、地点、组织、特殊要素。
- 本书激活的世界切片。
- 禁止方向与设定边界。

生成输出要求：

- 必须区分“全量世界资料”和“本书当前可用切片”。
- 必须产出硬规则、软规则、禁用方向、可扩展边界。
- 必须说明哪些设定来自强证据，哪些是 AI 合理补全。
- 必须服务故事宏观规划、角色生成和章节写作，不做纯展示型百科。

上游需要提供：

- 世界设定证据检索。
- 世界规则抽取。
- 势力、地点、组织、特殊要素抽取。
- 与当前小说 premise 的相关性评分。
- 可用于故事规划的世界切片候选。

### 4.2 人物基础信息

目标：从外部知识库中自动生成本书角色阵容的基础信息，帮助新手快速得到可用人物，而不是只返回人物百科。

本项目正式资产建议落到：

- 角色基础信息。
- 角色叙事岗位。
- 角色关系初始状态。
- 角色红线与不可写崩点。
- 角色在本书中的读者收益。

生成输出要求：

- 必须区分“人物事实”“本书改造建议”“AI 补全”。
- 必须给出角色在本书中的功能，不只给姓名、身份、外貌。
- 必须能支持后续章节角色上下文包。
- 如果上游资料只覆盖局部样本，要明确 coverage。

上游需要提供：

- 人物实体抽取与合并。
- 角色关系抽取。
- 人物行为、口吻、欲望、弱点证据。
- 适配当前小说定位的角色阵容建议。
- 角色事实与可改造空间的边界说明。

### 4.3 写法引擎

目标：从外部知识库中的范文、拆书、作者样本或作品样本中自动生成可执行的写法资产。

本项目正式资产建议落到：

- 写法资产 `StyleProfile`。
- 写法合同 `StyleContract`。
- 适用题材和适用场景。
- 风格来源证据。
- 绑定建议。

生成输出要求：

- 只提炼可迁移的表达特征，不复制受保护作品表达。
- 必须区分表达层规则和结构层规则。
- 不允许写法资产接管剧情结构、角色事实或世界事实。
- 必须能被写作、审校、修复链路消费。

上游需要提供：

- 写法样本检索。
- 文风、叙事距离、语言质地、对白方式、节奏密度抽取。
- 可迁移特征与不可迁移特征区分。
- 样本覆盖范围和风险提示。
- 与当前小说题材、目标读者、推进模式的适配判断。

### 4.4 反 AI 引擎

目标：从资料和目标写法中自动生成“去 AI 味”的规则组合，作为写法资产的一部分或独立规则绑定。

本项目正式资产建议落到：

- 反 AI 规则选择。
- 反 AI 检测重点。
- 高风险表达类型。
- 修复建议策略。

生成输出要求：

- 不做泛化的“不要像 AI”空话。
- 必须输出可检测、可修复的具体规则。
- 必须能结合当前写法判断哪些 AI 味是高风险。
- 必须允许不同题材、不同写法有不同的反 AI 重点。

上游需要提供：

- 样本文本的 AI 味风险抽取。
- 常见模板痕迹、总结腔、解释过量、情绪空转、机械排比等问题分类。
- 与当前写法资产匹配的反 AI 规则推荐。
- 检测依据和修复优先级。

### 4.5 文笔相关

目标：把“文笔”拆成系统可执行的表达维度，而不是让用户面对模糊审美词。

本项目正式资产建议落到：

- 语言质地。
- 句式节奏。
- 段落密度。
- 情绪外显方式。
- 感官描写密度。
- 叙事距离。
- 对白锋利度。
- 解释密度。

生成输出要求：

- 文笔不是独立漂浮资产，优先并入写法合同。
- 需要能按书级、卷级、章节级、角色视角级绑定。
- 必须能转成 writer prompt block、review rule、repair instruction。

上游需要提供：

- 文笔维度抽取。
- 样本句式与段落节奏统计或结构化描述。
- 可执行改写建议。
- 与目标小说的适配和风险说明。

## 5. 建议补充的资产类型

### 5.1 题材与读者承诺

原因：新手最容易卡在“这本书卖什么、前 30 章给什么承诺”。外部知识库如果能提供同类作品和题材资料，应辅助生成书级定位。

本项目正式资产建议落到：

- 题材基底。
- 目标读者。
- 核心卖点。
- 前 30 章承诺。
- 禁止误导的期待。

### 5.2 故事推进模式

原因：世界观和人物都需要服务“这本书靠什么持续推进”。如果只补设定不补推进模式，新手仍然难以写完整本。

本项目正式资产建议落到：

- 主推进模式。
- 副推进模式。
- 章节奖励类型。
- 冲突触发方式。
- 中后期可持续性风险。

### 5.3 故事宏观规划素材

原因：外部知识库不应直接替本项目写完整故事，但可以提供可用素材，帮助生成 premise、主冲突、悬念盒、成长路径和 payoff 种子。

本项目正式资产建议落到：

- premise 候选。
- core conflict 候选。
- mystery source。
- pressure source。
- payoff seeds。
- forbidden plot directions。

### 5.4 章节与卷级素材种子

原因：绑定资料如果只在开书时用一次，后续章节仍会发散。需要把可复用素材沉淀成卷级和章节级可调用种子。

本项目正式资产建议落到：

- 卷级 setpiece seeds。
- 章节任务素材。
- 场景素材。
- 冲突升级素材。
- 回收节点素材。

### 5.5 平台与合规边界

原因：某些知识库来源可能带有题材、平台、版权、敏感表达或不适配网文读者的风险。本项目需要在正式资产前记录风险。

本项目正式资产建议落到：

- 禁用素材。
- 高风险设定。
- 改造建议。
- 平台表达边界。
- 不可直接模仿的来源提示。

### 5.6 质量评估基线

原因：外部知识库生成的资产需要被后续审校和修复消费。应同时生成一套“这本书写得对不对”的评价基线。

本项目正式资产建议落到：

- 书级质量目标。
- 风格一致性检查点。
- 角色一致性检查点。
- 世界规则检查点。
- 章节偏航风险。

## 6. 默认生成流程

建议流程如下：

1. 用户在小说对象上绑定外部知识库来源。
2. 本项目发起资产生成任务，选择要生成的资产类型。
3. 上游返回候选资产、证据引用、覆盖范围和可信度。
4. 本项目将结果保存为 `GeneratedBookAssetDraft`。
5. AI 在本项目内进行二次适配，确保结果符合当前书级目标、已有世界、已有角色和写法边界。
6. 用户看到低认知负担的确认界面：推荐应用、需要确认、冲突风险、可忽略。
7. 用户确认后，草稿应用为本项目正式资产。
8. 后续规划、写作、审校、修复消费正式资产，并保留来源追溯。

## 7. 更新与冲突策略

### 7.1 外部资料更新

当外部知识库版本变化时，本项目不应静默覆盖已确认资产。

建议策略：

- 新版本进入差异草稿。
- 显示新增、删除、冲突和建议修改。
- 已被用户编辑过的本地资产优先级高于上游新建议。
- 允许用户选择只更新部分资产类型。

### 7.2 与本项目已有资产冲突

冲突优先级建议：

1. 用户明确确认过的本地资产。
2. 已进入章节正文和 canonical state 的事实。
3. 当前书级合同、世界规则、角色事实。
4. 外部知识库生成的新建议。
5. 模型自由补全。

外部建议如果与已写正文冲突，只能进入“建议修订”或“重规划候选”，不能直接改正式事实。

## 8. 上游接口能力清单

后续跨系统交互设计时，上游至少需要支持这些能力：

1. 来源绑定：按小说绑定外部资料集、作品、样本或分析结果。
2. 版本快照：返回稳定的 source version，避免生成过程读到变化中的资料。
3. 资料检索：按资产类型、问题、上下文进行语义检索。
4. 结构化抽取：世界观、人物、写法、反 AI、文笔、题材承诺、推进模式等分类输出。
5. 证据引用：每条关键判断都能返回可追溯 evidence refs。
6. 覆盖范围：明确资料覆盖的是全书、局部章节、样本文本还是单一资料包。
7. 适配推荐：结合本项目传入的小说上下文给出推荐，而不是只返回资料摘要。
8. 冲突提示：识别上游资料之间、上游资料与本项目上下文之间的冲突。
9. 增量更新：支持基于 source version 返回差异候选。
10. 任务化生成：长耗时抽取需要任务 ID、进度、失败原因和可恢复能力。

## 9. 本项目侧验收标准

1. 小说可以绑定外部知识库来源，并选择要生成的资产类型。
2. 世界观、人物、写法、反 AI、文笔至少能形成草稿并保存来源证据。
3. 生成结果必须落成本项目正式资产后才进入主创作链。
4. 用户能看到推荐应用项、需确认项、冲突项和资料覆盖范围。
5. 外部资料更新不会静默覆盖用户已确认资产。
6. 后续规划、写作、审校、修复能读取应用后的正式资产。
7. 所有核心判断都能追溯到上游证据或本项目 AI 补全原因。

## 10. 待确认问题

1. 外部知识库的最小上游形态是什么：资料检索服务、拆书结果服务、作品资产服务，还是三者都有？
2. 本期最小闭环是否只覆盖“单本小说绑定一个外部来源”，多来源合并放到下一期？
3. 世界观和人物是否允许直接创建新本地资产，还是必须先进入草稿确认？
4. 写法、反 AI、文笔是否作为一套写法资产保存，还是拆成独立资产后再绑定？
5. 上游是否已经能提供稳定版本号和 evidence refs，如果没有，本项目是否需要先设计临时快照层？
