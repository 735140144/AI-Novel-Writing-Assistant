# Fanqie Dispatch API Notes

Source: `/tmp/gelsangdispatch-release/docs/api.md`

## Integration Summary

The external service exposes Fanqie credential login, draft creation, publish execution, job status, and audit query APIs.

Base URL: `https://dispatch.lucky37.cn`

The documented flow is:

1. `POST /credentials` creates a credential and returns `credential.uuid`.
2. `POST /credentials/:credentialUuid/login-bootstrap` starts QR login.
3. The product displays `qrPageUrl`, `qrImageUrl`, `qrCodeBase64Png`, or terminal QR data to the user.
4. `POST /credentials/:credentialUuid/validate` verifies whether the credential is ready.
5. `POST /publish/jobs` submits draft or publish jobs.
6. `GET /jobs/:jobId` checks execution status.
7. If `CREDENTIAL_RELOGIN_REQUIRED` is returned, the user must scan QR again.

## Important API Constraints

* There is currently no business-level API authentication in the dispatch service. This project should call it from the server side and protect access through this project's own authenticated routes and ownership checks.
* `credentialUuid` is the primary Fanqie credential selector.
* `POST /publish/jobs` requires `credentialUuid`, `bookId`, `bookTitle`, `mode`, optional `requestId`, optional `publishOptions`, and `chapters`.
* `mode` is `draft` or `publish`.
* `publishOptions.useAi` is supported.
* `publishOptions.timerTime` supports `YYYY-MM-DD HH:mm`.
* `publishOptions.dailyWordLimit` is supported.
* Chapter content is sent only in the publish request body. The dispatch service states it does not persist chapter body content in job or audit records.
* Job status enum is `queued`, `leased`, `running`, `completed`, `failed`.
* Credential status enum is `created`, `login_pending`, `ready`, `expired`, `invalid`.
* Common relogin error code is `CREDENTIAL_RELOGIN_REQUIRED`, with a `relogin.action` of `bootstrap_login`.

## Product Mapping Notes

* Local platform binding should store the user's platform credential UUID and credential status.
* Local novel-platform binding should store the target external `bookId`, `bookTitle`, and selected platform.
* Publish records should keep local chapter status independent of dispatch job status, because the product needs user-facing states such as unpublished, draft box, and published.
* Dispatch job status should be stored as execution telemetry for each submitted batch.
* `requestId` should be generated by this project for idempotency and traceability. A stable local publish item or batch ID is a good source.
* A full "daily 08:00, 2 chapters per day" plan should be computed locally. Submission should group chapters by identical planned publish time and pass `publishOptions.timerTime` as `YYYY-MM-DD HH:mm`.

## Open API Gaps

* No documented endpoint for listing Fanqie books under a credential.
* No documented endpoint for checking whether a chapter already exists in the platform draft box.
* No documented per-chapter scheduled publish timestamp field inside one job. If chapter times differ, submit separate jobs grouped by planned time.
* No documented webhook/callback for job completion; polling `GET /jobs/:jobId` is the available path.
