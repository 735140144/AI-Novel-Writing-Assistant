# 新增发布平台

## Goal

为小说新增发布平台能力，让用户可以把本项目中的小说绑定到外部小说平台账号和平台书籍，按发布计划把章节提交到指定平台，并在本项目内看到每章从未发布、草稿箱到已发布的状态。

在最新优化方向里，发布平台不再只是一个单页工作台，而是拆成菜单级发布模块：提供账号管理、作品列表和作品级发布详情页，降低新手在账号资产、作品绑定和具体发布操作之间来回切换的混乱感。

这条能力服务于新手完整成书后的发布闭环：用户不需要自己维护发布时间表、逐章复制正文、判断哪些章节已经提交。系统应提供清晰默认值、自动生成计划、状态追踪和失败恢复入口。

## What I Already Know

* 用户希望新建独立 Trellis task。
* 目标功能包括：绑定小说平台、将小说发布到指定小说平台、生成发布计划、调用发布接口时带入发布时间、展示发布状态。
* 示例节奏是“每日 8 点发布 2 章节”，系统需要据此生成每章发布时间。
* 用户要求状态至少包含：未提交到平台时为未发布，提交到平台草稿箱时为草稿箱，已发布时为已发布。
* 外部发布接口参考 `/tmp/gelsangdispatch-release/docs/api.md`。
* 参考 API 当前是 Fanqie Draft Publisher API，base URL 为 `https://dispatch.lucky37.cn`。
* 参考 API 支持创建凭据、二维码登录、校验凭据、提交草稿/发布任务、查询 job、查询 audit。
* 参考 API 没有业务级鉴权，应由本项目服务端封装调用，不能让前端直接访问外部服务。
* 参考 API 使用 `credentialUuid` 选择番茄凭据，`bookId` / `bookTitle` 指定平台书籍。
* 参考 API `POST /publish/jobs` 的 `mode` 为 `draft` 或 `publish`。
* 用户已确认发布接口支持 `YYYY-MM-DD HH:mm` 格式的发布时间。
* 参考 API job 状态为 `queued`、`leased`、`running`、`completed`、`failed`。
* 参考 API 凭据状态为 `created`、`login_pending`、`ready`、`expired`、`invalid`。
* 本项目已有小说、章节、用户归属、章节管理、导出和任务中心基础。
* 现有 `Novel` 与 `Chapter` 已按 `userId` / `novelId` 建立归属关系；新增发布平台数据也应保持用户与小说隔离。
* 用户最新要求将发布平台进一步拆分成菜单栏里的发布模块。
* 发布模块至少拆成：账号管理、作品列表、作品级发布页面。
* 账号管理负责绑定账号和管理账号资产，而不是和作品发布操作混在同一页。
* 作品列表需要支持添加作品、绑定账号、进入具体发布页面。
* 作品级发布页面需要承载发布计划、发布状态，以及对已有书籍已发布/草稿箱状态的同步。
* 用户明确要求发布时一次只提交一章，避免一次性发布多章。

## Assumptions

* MVP 先接入番茄平台，平台能力用通用发布平台模型承载，后续可以扩展其他平台。
* 平台账号绑定与本项目用户绑定，平台书籍绑定与本项目小说绑定。
* 发布计划由本项目保存；外部发布服务负责接收逐章提交任务并返回执行状态。
* 用户输入自然语言发布节奏时，使用 AI 结构化理解生成计划参数，再由确定性逻辑计算每章发布时间；不能用关键词/正则分支作为产品核心解析路径。
* 外部服务不保存章节正文，但本项目本来已保存章节正文；本项目的发布记录不需要重复持久化完整正文快照，除非后续明确需要审计级发布快照。
* 外部发布接口的 `publishOptions.timerTime` 按 `YYYY-MM-DD HH:mm` 传入。
* 当前 dispatch 侧已补充 `GET /credentials/:credentialUuid/books/progress` 只读进度接口，可返回 `publishedChapters`、`draftChapters` 和过滤后的 `effectiveDraftChapters`，供本项目在生成计划前做远端去重与续排。

## Open Questions

* 当前无阻塞性开放问题；进入实现前需要把上述确认项落到路由、数据模型和 dispatch 同步接口设计。

## Requirements

### 1. 菜单与信息架构

发布能力需要从单页工作台继续拆分成菜单级发布模块。

菜单层级至少应满足：

* 顶层存在“发布”栏目。
* “发布”栏目下至少包含“账号管理”“作品列表”两个入口。
* 作品列表中的单条绑定记录需要能进入独立的作品级发布页面。
* 账号管理、作品列表、作品级发布页面的职责边界需要清晰，避免在同一页混合过多动作。
* 作品级发布页面的路由和数据加载需要以“发布绑定记录”为中心，而不是继续假设一本小说只对应一个平台绑定。

### 2. 平台账号绑定与资产管理

用户可以在本项目内创建或绑定小说平台账号。

MVP 至少支持番茄平台：

* 创建平台凭据，保存 `credentialUuid`、平台类型、账号标签、凭据状态、最后校验时间。
* 发起二维码登录，向用户展示扫码入口。
* 校验登录状态，展示 ready / pending / expired / invalid 等状态。
* 当发布接口返回 `CREDENTIAL_RELOGIN_REQUIRED` 时，引导用户重新扫码。
* 当扫码校验成功并进入 `ready` 状态后，二维码需要自动消失，避免继续展示过期扫码入口。
* 当扫码校验成功并拿到平台账号显示名后，账号标签需要自动同步为番茄返回的笔名/账号显示名，减少用户手工维护。
* 账号管理页需要展示账号资产清单和状态，而不是只在作品页内临时选择账号。

### 3. 作品列表、绑定关系与作品级入口

用户可以把本项目中的一本小说绑定到指定平台书籍。

绑定信息至少包括：

* 本项目 `novelId`。
* 平台类型。
* 平台凭据。
* 平台 `bookId`。
* 平台 `bookTitle`。
* 绑定状态和最近同步/校验时间。

MVP 交互要求：

* 绑定番茄书籍时，本地交互优先使用下拉选择，而不是直接暴露原始 `bookId` 输入框作为主操作。
* 用户视角应优先看到“本地维护的番茄书名”，系统保存后再自动带出对应的 `bookId` / `bookTitle` 调用接口。
* 如果当前文档能力不足以从番茄侧自动拉取书籍列表，则由本项目维护每本小说对应的番茄书籍映射，再通过下拉供用户选择；不要把“番茄 ID 是什么”作为主要用户概念暴露。
* “添加作品”采用混合模式：先从现有小说库中选择加入；加入并完成账号绑定后，这本书会长期出现在发布模块的作品列表里进行后续管理。
* “添加作品”后，如果尚未绑定任何账号，则不进入作品列表；只有形成“小说 x 账号绑定记录”后才显示在列表中。
* 作品列表的数据行粒度是“一本小说 x 一个绑定账号”。如果同一本小说绑定 2 个账号，则列表中出现 2 条记录。
* 同一条作品记录需要同时承载：小说、平台账号、平台书籍三者之间的绑定关系。
* 作品列表不要使用卡片展示，MVP 采用表格或紧凑列表展示，便于扫描多条绑定记录。
* 作品列表至少展示以下列：
  * 书名
  * 进度（至少包含已发布章节数 / 当前已完成章节数 / 预期章节数）
  * 发布平台
  * 发布平台笔名
  * 发布书名（平台侧书名）
  * 账号状态
  * 最近同步时间
  * 详情入口按钮
* 如果部分推荐字段在当前实现阶段拿不到，可以降级隐藏，但以上字段是优先展示顺序。
* 详情入口按钮文案暂定为“发布详情”，用于进入该绑定记录对应的作品级发布页面。

### 4. 作品级发布页面与计划生成

用户可以输入发布节奏，例如“每日 8 点发布 2 章节”。

系统需要生成结构化发布计划：

* 起始日期或起始章节。
* 参与发布的章节数量。
* 每日发布章节数。
* 计划发布时间。
* 参与发布的章节范围。
* 每章计划发布时间，格式为 `YYYY-MM-DD HH:mm`。
* 计划状态。

如果用户只输入节奏而没有给起始日期，系统应给出低认知负担默认值，例如从用户确认计划后的下一天开始。

发布计划还需要支持“参与发布章节数量”输入：

* 默认值为 `当前已完成章节数 - 已发布章节数`。
* 用户可以人工修改该数量，例如改为 `10`，表示从当前已发布进度之后按顺序选取接下来的 10 章参与本次计划。
* 系统需要基于这个数量和当前远端校验后的发布进度，确定本次计划覆盖的章节范围。

生成计划前必须先检查当前平台绑定的远端真实进度，并结合本地计划占用情况续排：

* 远端已进入草稿箱或已发布的章节，不得再次生成重复待提交流程项。
* 如果已有未完成计划，新的发布时间表应从最后一个已占用发布时间之后继续排，不覆盖已提交章节。
* 续排逻辑需要同时参考远端同步结果、本地已提交 job、章节状态和计划项占用情况。
* 作品级发布页的数据主键应优先使用 `bindingId`，这样同一本小说绑定多个账号时，每条记录都有独立计划、状态和提交流程。

### 5. 章节发布状态与状态同步

每章需要有独立的平台发布状态。

面向用户的核心状态至少包括：

* 未发布：尚未提交到平台。
* 草稿箱：已提交到平台草稿箱。
* 已发布：已提交发布并确认完成。

系统内部还应能记录提交中、失败、需要重新登录、外部 job 状态、错误信息和最后提交时间，用于恢复与排错。

对已有书籍，作品级发布页面还需要支持同步或刷新下列状态：

* 已发布章节。
* 已进入平台草稿箱的章节。

用户已明确要求生成发布时间表前必须先校验远端真实进度，因此首版不能只依赖本地事实源续排。

MVP 同步要求调整为：

* 生成发布时间表前，必须先读取当前绑定记录在平台侧的真实进度。
* 远端已发布章节、已存在草稿章节都需要纳入去重判断。
* 远端同步优先使用 dispatch 的 `effectiveDraftChapters` 作为“有效草稿”集合，忽略空白草稿、`新建章节` 和类似 `第4章` 的占位草稿。
* 如果远端同步失败、登录失效或结果不可信，则阻止生成新计划，并明确提示用户先完成重新登录或重试同步。
* 本地状态仍然需要保存同步结果和最后同步时间，但不再把“本地已知进度”当成唯一事实源。
* 同步策略采用手动触发为主，不在进入详情页时自动发起远端同步。
* 如果当前绑定记录还没有生成过发布时间表，则在首次生成计划前必须先完成一次远端同步。
* 作品级发布页需要提供明确的“同步远端进度”入口，成功后才允许首次生成发布时间表。
* 调用远端同步接口时，需要传入绑定记录对应的 `credentialUuid`、`bookId`、`bookTitle`；如果 dispatch 返回 `409 CREDENTIAL_RELOGIN_REQUIRED`，本项目应将其转译为重新扫码提示并阻止后续计划生成。

### 6. 发布执行

用户可以按计划提交章节到指定平台。

执行要求：

* 从本项目读取章节标题、正文、章节序号和卷名。
* 调用外部 `POST /publish/jobs`。
* 生成稳定 `requestId`，支持幂等追踪。
* 传入 `credentialUuid`、`bookId`、`bookTitle`、`mode`、`publishOptions` 和章节列表。
* `publishOptions.timerTime` 必须使用 `YYYY-MM-DD HH:mm`。
* 发布计划中的时间仅用于生成并传给外部接口，本项目不负责在本地等到某个时间点后自动触发提交。
* 用户发起发布后，系统按发布计划中的章节顺序逐章调用外部接口。
* 每次接口调用只提交一章，避免一次性发布多章。
* 点击一次“开始发布”后，系统会按本次计划顺序连续逐章提交；如果中途某一章失败，则停止后续提交，等待用户处理后再继续。
* 调用后保存 dispatch `jobId`、job 状态、提交批次、章节映射和错误信息。
* 通过 `GET /jobs/:jobId` 查询任务结果并更新本地状态。

### 7. UI / Workflow

发布能力应提供菜单级独立入口，不要嵌入小说列表卡片，也不要放在小说编辑页顶部按钮区。

当前目标流程应从“单页选小说后完成全部操作”演进为分层流程：

* 进入“账号管理”绑定和管理账号资产。
* 进入“作品列表”添加作品、查看绑定情况、进入具体作品发布页。
* 在作品级发布页面里处理发布计划、章节状态、同步状态和提交动作。

小说工作台内部已有靠近成书完成环节的“发布”步骤能力应保留，供工作台流程继续使用，但主入口应从菜单进入。

UI 文案需要从用户视角描述下一步，例如“绑定番茄账号”“生成发布时间表”“提交到草稿箱”，避免描述实现过程。

## Confirmed Decisions

* 发布时间使用 `YYYY-MM-DD HH:mm` 格式传给外部发布接口。
* 发布计划在本项目内保存到每章。
* 发布执行按计划顺序逐章调用接口；计划时间只传给外部接口，不由本项目本地定时触发。
* 每次接口调用只提交一章，避免一次性发布多章。
* 点击一次“开始发布”后，会按计划顺序连续逐章提交；遇到失败则停止后续提交。
* “添加作品”采用混合模式：从现有小说中选择加入，加入后在发布模块里长期管理。
* 作品列表按“小说 x 绑定账号”展示；同一本小说绑定多个账号时，列表展示多条记录。
* 作品列表使用表格/列表展示，不使用卡片。
* 列表详情按钮文案暂定为“发布详情”。
* 生成发布时间表前，必须以远端真实进度为准，不能只靠本地记录续排。
* 远端同步采用手动触发策略；首次生成发布时间表前必须先同步一次。
* 未绑定账号的小说不进入作品列表；作品列表只展示已形成绑定关系的记录。
* 发布计划需要包含“参与发布章节数量”输入，默认值为“当前已完成章节数 - 已发布章节数”，且允许人工修改。
